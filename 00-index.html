信息架构：从用户旅程到章节结构

本手册的章节不是凭感觉排的，而是从用户旅程地图里的 8 项关键任务（T0–T7）推导出来的。下面用层次任务分析（HTA, Hierarchical Task Analysis）的全景树展示所有用户任务如何分 3 个阶段，以及每项任务对应到哪一个章节。

从全景树可以看到一条清晰的读者推进路径：阶段 A 建立心智模型与跑通最小闭环（第 1–2 章）→ 阶段 B 掌握主流上手用法（第 3–5 章）→ 阶段 C 形成兜底保障（第 6–8 章）。每项用户任务内部还会继续拆到具体操作步骤，那才是各章节小节标题的来源，全景树只是把「任务 → 章节」的顶层连线可视化出来。8 个章节恰好对应 8 项用户任务，没有遗漏、也没有冗余。

写作风格指南（Style Guide）

给所有组员的“共同纪律”（本手册所有文档必须遵守）:
  1. 第二人称：全文用“你”，不混用“您/用户/操作人员”
  2. 任务标题动词开头：“安装 Claude Code CLI” ✅ | “Claude Code 的安装” ❌
  3. 术语一致：一个概念全文一个译名（详见附录 B 术语表）
  4. 提示框四分色：note（蓝·上下文）/ tips（绿·技巧）/ warning（黄·易错）/ danger（红·危险）
  5. 代码用 <code> 或 <pre> 包裹，不要在正文里混排裸代码
  6. 中文标点：中文语境内用全角标点（, 。 ; : ( ) " " ? !），英文代码、URL、文件路径内保留半角
  7. 每章末尾：放“读完本章你应该能做”的 ✅ 自测清单

排版约定

  • 行内代码 —— 命令、文件名、配置项、函数名
  • 蓝色提示框 —— 需要注意但不紧急的信息，一般是上下文补充
  • 绿色提示框 —— 小技巧、最佳实践、正向引导
  • 黄色警告框 —— 高频易错点，跳过会卡住
  • 红色危险框 —— 涉及数据安全、不可逆操作，务必阅读
