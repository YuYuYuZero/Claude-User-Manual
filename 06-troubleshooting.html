6.1 安装与配置类

安装
6.1.1 · 症状：npm install -g @anthropic-ai/claude-code 报 EACCES 权限错误

可能原因：用了系统自带 Node，全局目录（如 /usr/local/lib/node_modules）需要 root 权限写入。

解决方案：不要用 sudo 硬装，改用 nvm 管理 Node:

brew install nvm
mkdir -p ~/.nvm
echo 'export NVM_DIR="$HOME/.nvm"' >> ~/.zshrc
echo '[ -s "$(brew --prefix nvm)/nvm.sh" ] && . "$(brew --prefix nvm)/nvm.sh"' >> ~/.zshrc
source ~/.zshrc
nvm install --lts
预防措施：团队约定用 nvm，把 .nvmrc 提交到 Git，新人一行 nvm use 拉齐环境。
安装
6.1.2 · 症状：设了环境变量，但 VS Code 扩展不生效

可能原因：VS Code 不会实时继承刚改的 ~/.zshrc。它启动时读的是它自己启动那一刻的环境变量快照。

解决方案：

  1. 完全退出 VS Code（Cmd+Q，不是关窗），重新打开
  2. 或者先在终端 source ~/.zshrc，再用 code . 启动 VS Code
预防措施：每次修改 ~/.zshrc 后，形成“关 VS Code → 重开”的条件反射。
配置
6.1.3 · 症状：配置了第三方 API 代理，但请求一直 401

可能原因（按概率排序）:

  • 原因 A:ANTHROPIC_BASE_URL 写成了 https://api.xxx.com/v1（多了 /v1）
  • 原因 B:ANTHROPIC_AUTH_TOKEN 值前后有多余空格或换行
  • 原因 C：代理的 Key 已过期或被禁用

解决方案：

  1. 检查 ~/.claude/settings.json:Claude Code 侧不带 /v1
  2. 用 echo $ANTHROPIC_AUTH_TOKEN | cat -A 看是否有 $ 前的空格或 ^M 换行
  3. 去代理平台重新生成 Key，检查账户是否欠费
预防措施：把 Base URL 写进 CLAUDE.md 或团队 README，避免每个新人踩同一个坑。

6.2 API 调用类

API
6.2.1 · 症状：请求报错 "max_tokens: ... must be less than..."

可能原因：max_tokens 超过了模型的最大输出上限。例如 Sonnet 4.6 是 64k,Opus 4.7 是 128k,Haiku 4.5 是 64k。

解决方案：查附录 A 模型对照表，把 max_tokens 调到模型支持范围内。

相关提示：扩展思考时 budget_tokens 必须 < max_tokens，两者都计入模型上限。

API
6.2.2 · 症状：用 Claude Opus 4.7 调手动扩展思考，返回 400 错误

可能原因：Claude Opus 4.7 不支持手动扩展思考（thinking={"type":"enabled"}）。

解决方案：二选一：

  • 改用自适应思考(adaptive thinking)——让模型自己决定思考深度，无需 budget_tokens
  • 或改用支持手动思考的模型，如 Claude Sonnet 4.5 等
预防措施：在调用前先用附录 A确认模型是否支持你要用的特性。
API
6.2.3 · 症状：用 Claude Opus 4.7 / 4.6 / Sonnet 4.6 预填充 assistant 消息返回 400

可能原因：这三个模型明确不支持预填充技术——在 messages 最后一位放 assistant 消息会直接报错。

解决方案：

  • 改用“结构化输出”（response_format / 严格工具使用）
  • 或改写系统提示，用指令约束输出格式
  • 或改用支持预填充的模型（如 Sonnet 4.5）
缓存
6.2.4 · 症状：提示缓存第二次请求没命中，成本还是很高

可能原因：

  • 原因 A：扩展思考的 budget_tokens 变了 —— 消息缓存会失效（系统提示缓存仍有效）
  • 原因 B:5 分钟缓存已到期，两次请求间隔过长
  • 原因 C：两次请求的 system 或前缀内容有细微差异（包括空格、换行）

解决方案：

  1. 保持思考参数一致——只在必要时改 budget_tokens
  2. 如果需要跨会话复用，开启 1 小时缓存（比 5 分钟缓存更贵一点，但命中率高得多）
  3. 检查 system 字符串是否完全一致
工具
6.2.5 · 症状：工具使用时报 "You must include the thinking block"

可能原因：扩展思考 + 工具使用时，第二次请求必须把完整的 thinking 块连同 tool_use 块一起传回，不能修改或剥离。系统用密码签名验证思考块真实性。

解决方案：

# 第二次请求必须原样带回思考块和工具块
continuation = client.messages.create(
    ...
    messages=[
        {"role": "user", "content": "What's the weather in Paris?"},
        # 下面这一整块必须原样回传,不能改
        {"role": "assistant", "content": [thinking_block, tool_use_block]},
        {"role": "user", "content": [
            {"type": "tool_result",
             "tool_use_id": tool_use_block.id,
             "content": "Current temperature: 88°F"}
        ]}
    ],
)
预防措施：代码里建立一个“思考块 → 对应工具调用”的映射结构，方便原样回传。
工具
6.2.6 · 症状：响应返回 stop_reason: "tool_use"，但我不知道下一步传什么

可能原因：第一次写工具调用的代码，没理解“工具循环”的流程。

解决方案：完整流程是：

  1. 你发第一个请求（带 tools 参数）
  2. Claude 返回 stop_reason: "tool_use" + 一个或多个 tool_use 块
  3. 你的代码执行这些工具（读文件 / 调 API / 跑脚本...）
  4. 你把完整的对话历史 + 每个 tool_use 对应的 tool_result一起发第二次请求
  5. Claude 看到结果后，要么再调工具（回到步骤 3），要么返回 stop_reason: "end_turn"
预防措施：先用一个简单的服务器端工具（如 web_search）练手——服务器端工具 Anthropic 替你跑，你直接拿结果，跳过了步骤 3-4 的复杂流程。
API
6.2.7 · 症状：输出明明还没说完就停了，stop_reason: "max_tokens"

可能原因：max_tokens 设得太小，模型被截断。

解决方案：

  1. 把 max_tokens 调大（注意不能超过模型上限，见附录 A）
  2. 如果是扩展思考场景，记得 budget_tokens 占用的是 max_tokens 预算——两者都要调
  3. 若要继续上次未完成的输出，把第一次的截断响应作为 assistant 消息，追加一个 user 消息“继续”
API
6.2.8 · 症状：返回 overloaded_error 或 529

可能原因：服务端瞬时过载，不是你的问题。

解决方案：

  1. 实现指数退避重试：等 1 秒重试，失败等 2 秒，再失败等 4 秒，最多重试 3–5 次
  2. 如果是长期批量任务，考虑用批处理 API（比标准 API 便宜 50%）
  3. 错峰发送：在高峰时段延后请求
预防措施：生产环境必须写好重试和降级逻辑，不要假设 API 永远可用。

6.3 Claude Code 使用类

Skill
6.3.1 · 症状：Skill 装了但 Claude 好像没调用它

可能原因：

  • 原因 A:description 写得太抽象，Claude 判断不出何时该触发
  • 原因 B：安装后没有重启 Claude Code
  • 原因 C：在 API 场景用预构建 Skills 时，没加齐所需 beta 标头

解决方案：

  1. 改写 description——用用户可能说的原话，如“帮我填写 PDF 表单”、“分析 Excel 销售数据”
  2. 重启 Claude Code
  3. 如果自动触发不成功，手动指定：对话里直接说“用 xxx skill 帮我...”
权限
6.3.2 · 症状：Claude 总是说“我不能做这个”或拒绝执行某条命令

可能原因：

  • 原因 A：你在 default 权限模式下，它需要你按 y 确认——它不是“拒绝”，是在等你点头
  • 原因 B:Opus 4.7 的网络安全护栏触发了（如批量删文件、敏感数据外泄等）
  • 原因 C：命令本身确实危险（rm -rf / 这类），被安全规则硬拦截

解决方案：

  1. 按 Shift+Tab 看屏幕底部状态栏——是不是显示 "default mode"？如是，按 y 确认即可
  2. 如果是合法的安全研究/渗透测试，把上下文（项目目的、合法授权）写进 CLAUDE.md
  3. 对于真的危险操作，应该庆幸它拒绝了——换个更安全的实现方式
登录
6.3.3 · 症状：Claude 突然要求重新登录，或 claude 命令报“未授权”

可能原因：账号 token 过期。Claude 长时间没用会自动失效。

解决方案：

claude logout
claude login
# 浏览器会自动打开,完成授权即可
预防措施：无需特别预防——失效是正常安全机制，平均 1–2 个月可能要重新登录一次。
MCP
6.3.4 · 症状：MCP 服务器一直连接不上，工具列表里看不到对应工具

可能原因（按概率）:

  • 原因 A：对应服务的认证过期（GitHub token、Slack OAuth 等）
  • 原因 B：网络问题，公司防火墙或代理拦截了
  • 原因 C:MCP 服务器本身挂了，稍等再试

解决方案：

  1. 先看状态：/mcp
  2. 如果显示 auth 错误，重新授权对应服务
  3. 试试 curl <mcp-server-url>，排除网络问题
  4. 实在不行，先 /mcp remove <id> 再 /mcp add <url> 重连
命令
6.3.5 · 症状：写好了 .claude/commands/foo.md，但 /foo 不生效

可能原因：

  • 原因 A：文件路径不对——必须在 .claude/commands/（项目级）或 ~/.claude/commands/（个人级）下
  • 原因 B：文件名后缀不是 .md
  • 原因 C:YAML frontmatter 格式坏了（三个 --- 必须正确成对）
  • 原因 D：加完命令没有重启 Claude Code

解决方案：

  1. 检查路径：ls -la .claude/commands/ 或 ls -la ~/.claude/commands/
  2. 检查文件名以 .md 结尾
  3. 检查 frontmatter：开头 --- 一行，结尾 --- 一行，中间是 YAML
  4. 退出 Claude Code（/quit）再重新启动
  5. 启动后输入 /help 确认命令是否在列表里

6.4 性能类

性能问题几乎都和“上下文越长，每次处理越慢、越贵、越容易出错”有关。核心策略是勤用 /compact 和 /clear。

性能
6.4.1 · 症状：聊了很久之后，Claude 开始“答非所问”，或者忘了最开始的设定

可能原因：上下文窗口快满了，出现“上下文衰退（context rot）”。

解决方案：

  1. 输入 /compact，让 Claude 主动总结前文、释放空间
  2. 如果当前子任务已完成，直接 /clear 清空；把关键结论记到 CLAUDE.md 里，下次启动自动加载
  3. 关掉再回来：claude --resume 或简写 claude -c 接续上次对话
预防措施：长任务不要一把梭——分阶段做，每完成一个阶段就 /compact，大任务拆成多个会话。
性能
6.4.2 · 症状：Claude Code 执行某个操作卡住不返回

可能原因：长任务超时（默认 API 超时受 API_TIMEOUT_MS 控制）。

解决方案：

  1. 按 Ctrl+C 中断
  2. 在 ~/.claude/settings.json 里把 API_TIMEOUT_MS 调大（默认建议 3000000 = 50 分钟）
  3. 把大任务拆成多个小步骤，每步让 Claude 先汇报再继续——既容易调试也容易回滚
性能
6.4.3 · 症状：聊得越久，Claude 回复越慢

可能原因：上下文堆积。每条消息都要重新处理全部历史——消息越长、处理越慢、单次成本越高。

解决方案：

  1. 立刻 /compact 压缩
  2. 实在太乱，导出当前进度 /export progress.md，然后 /clear 开新会话，把 progress.md 读进来继续
  3. 用 /context 查看当前占用——彩色网格直观显示文件/对话历史/系统提示分别占了多少
预防措施：看到 /context 显示 > 80%，立刻 /compact。不要等系统提示 "context full" 才动手。

6.5 模型弃用与迁移类

弃用
6.5.1 · 症状：调用 claude-sonnet-4-20250514 或 claude-opus-4-20250514 返回废弃提示

可能原因：这两个模型已宣布将于 2026 年 6 月 15 日停用。Claude Haiku 3 也将于 2026 年 4 月 19 日停用。

解决方案：

  • 迁移到 Claude Sonnet 4.6 / Opus 4.7 / Haiku 4.5
  • 参见附录 A 的 API ID 对照
预防措施：不要在代码里硬编码带具体日期的 ID（如 claude-sonnet-4-20250514），优先用别名（如 claude-sonnet-4-0）。

6.6 如果你的问题不在上面

按下面顺序排查：
  1. 先看报错原文——它通常已经告诉你哪里错。别跳过
  2. 确认模型是否支持你要的功能——查附录 A
  3. 在 Claude Code 里问 Claude——把完整错误信息贴进去让它分析
  4. 减法排查——把请求简化到最小，逐步加回复杂度，找到第一个出问题的变量
