第五章 · Agent Harness

本章不讲怎么用 Claude，讲怎么让 Claude 不失控。这一步跨过去，你才能真正用 AI 完成“2 小时不间断的复杂任务”，而不是每 5 分钟手动救火一次。

5.1 一句话：Harness 是包在模型外的“操作系统”

核心共识：更强的模型 + 裸奔的环境 < 稍弱的模型 + 精心设计的 Harness。

很多人以为“模型越强、任务越能搞定”。但实际上，真正决定长任务成败的是模型外面那一层——Agent Harness。

5.2 类比：电脑是怎么运行的

电脑组件对应 AI 系统职责
CPUAI 模型（Claude Opus/Sonnet/Haiku）逻辑计算和推理
内存（RAM）上下文窗口（Context Window）装载短期记忆
操作系统（OS）Agent Harness管理内存、调度工具、安全拦截

没有 OS，CPU 算力再强也跑不起来完整软件。同理，没有 Harness，模型再强也做不完一个 2 小时的长任务。

5.3 没有 Harness 的 4 大痛点

假设你让 AI 去执行一个需要持续 2 小时的复杂任务——比如“重构整个项目里所有的网络请求”——不给它 Harness 会遇到什么？

1. 上下文腐烂（Context Rot）
跑到第 50 步，彻底忘记了最初的目标和一开始定的规则。
2. 失控死循环
遇到某个报错，无限重复同一种错误方案，直到 Token 耗尽。
3. 幻觉累积（流鞭效应）
第一步的小错误，在后续 100 步中被不断放大，最终结果完全不可用。
4. 无安全边界
自作主张删掉了生产数据，或把机密文件发给了外部客户。

5.4 一个完整 Harness 的 6 个核心模块

#模块它帮 AI 做了什么
1提示词管理(Prompt Management)每次启动/重置时，重新注入系统规则和目标，防止 AI 跑偏（类似开工前的每日晨会）
2记忆管理(Memory Management)Harness 的灵魂。长任务中自动“压缩”对话，剔除废话，只保留结构化关键结论
3工具调度(Tool Orchestration)把读写文件、网络搜索、API 调用等底层能力封装好，等待模型召唤
4生命周期控制(Lifecycle)监控运行时间、执行步数，发现死循环立刻强行打断
5安全守卫(Safety Guardrails)遇到高风险操作（如 rm -rf、发邮件）强制暂停，等人类确认
6监控与日志(Observability)记录每一步花掉的 Token 成本、工具调用入参出参，方便复盘

5.5 Claude Code 本身就是一个开箱即用的 Harness

你每次在 Claude Code 里按 y/n 确认写操作、用 /compact 压缩上下文、看 /cost 花了多少钱——这就是 Harness 在工作。

它背后对应平台的若干底层能力：

  • 服务器端压缩（Compaction）：接近窗口限制时，自动总结对话早期部分
  • 上下文编辑（Context Editing）：用可配置策略自动管理上下文（工具结果清除、思考块清除）
  • 提示缓存：把稳定前缀缓存下来，降低重复调用的成本
  • 内存工具（memory）：跨对话存储和检索信息，构建长期知识库

5.6 Harness 运转的 6 步标准流程（以处理邮件为例）

当 Harness 接管任务时，它的生命周期如下：

  1. 启动（Boot）:Harness 注入系统提示，告诉 AI:“你现在是客服主管，目标是处理这 100 封邮件，只准用查询和回复工具。”
  2. 感知（Perceive）:Harness 把第一封杂乱的邮件内容转成 AI 容易理解的 JSON 格式喂给模型
  3. 推理 + 行动（Reason & Act）:AI 决定查询知识库并起草回复，Harness 替它去调用数据库
  4. 安全检查（Guard）:AI 决定发送补偿方案，Harness 识别到“退款/补偿”是敏感行为，立刻暂停等人类确认
  5. 记忆更新（Memory Update）：处理完毕，Harness 把“已处理客户A，退款50元”记入摘要日志，清空刚刚处理这封邮件的几十轮废话记忆，为下一封邮件准备干净大脑
  6. 循环或完成（Loop or Done）：回到第 2 步处理下一封，直到 100 封处理完毕，输出最终报告

5.7 Harness 与 Skills 的神仙配合

Skill 是“说明书”，Harness 是“车间流水线”。
场景只用 Skill（轻装）把 Skill 装进 Harness（流水线）推荐
单次任务
一个 Excel、一份周报、一次查询		✅ 你让 AI 读一个 Excel、写一份周报——一次对话能搞定，Skill 提供“怎么读 xlsx”的说明书就够了⚠️ 过度设计。一个回合能跑完的事情上流水线，只会增加启动、调度、日志的额外开销，得不偿失👉 只用 Skill
批量任务
50 个项目、100 封邮件、整库扫描		❌ 跑 3 个文件后就开始混乱——上下文塞满、记不住前面处理过的结论、中间报错就全盘崩✅ “遍历这 50 个项目的周报文件夹，每个都用 xlsx Skill 提取数据、汇总成主报告”。Harness 负责：调度循环、控制内存不爆、出错重试👉 Skill + Harness

5.8 动手：一个 100 行 Python 的最小 Harness

下面是一个安全代码审计 Harness 的脱水示例。它做了 4 件事——加载规则、循环扫描、人工拦截、记忆压缩——展示了 Harness 的核心骨架：

import os
import json
from your_llm_sdk import call_llm  # 伪代码,替换成你的 LLM 调用

class SecurityHarness:
    def __init__(self):
        self.memory = []                                # 短期记忆池
        self.global_bug_report = ""                     # 长期记忆(压缩后的全局报告)
        self.files_processed_since_last_compaction = 0
        self.critical_files = ["auth.ts", "config.js", "db.py"]  # 敏感名单

    def load_skill(self):
        """机制一:加载专家 Skill"""
        with open("security-check.md", "r") as f:
            return f.read()

    def compact_memory(self):
        """机制四:记忆压缩(Harness 的最强武器)"""
        print("\n[Harness] 触发记忆压缩...")
        compaction_prompt = (
            f"请把以下散乱的审计记录,压缩为不超过 100 字的结构化漏洞摘要。"
            f"现有全局摘要:{self.global_bug_report}。"
            f"近期记录:{self.memory}"
        )
        new_summary = call_llm(prompt=compaction_prompt)
        self.global_bug_report = new_summary
        # 残忍地清空短期记忆,只保留全局摘要
        self.memory = []
        self.files_processed_since_last_compaction = 0
        print(f"[Harness] 压缩完成:{self.global_bug_report}")

    def human_in_the_loop_guard(self, file_name, ai_proposal):
        """机制三:人工安全拦截器"""
        if file_name in self.critical_files:
            print(f"\n⚠️  [Harness] AI 试图修改核心文件:{file_name}")
            print(f"AI 的修改建议:\n{ai_proposal}")
            user_input = input(">> 是否允许?(Y 放行 / N 跳过): ")
            return user_input.strip().upper() == "Y"
        return True

    def run(self, project_dir):
        """机制二:游标循环引擎"""
        skill_prompt = self.load_skill()
        for file_name in os.listdir(project_dir):
            print(f"\n-> Harness 调度 Agent 审计文件:{file_name}")

            with open(os.path.join(project_dir, file_name), "r") as f:
                code_content = f.read()

            system_context = (
                f"全局已知漏洞摘要:{self.global_bug_report}\n\n"
                f"专业技能要求:{skill_prompt}"
            )
            ai_response = call_llm(
                system=system_context,
                user=f"请审计此代码:\n{code_content}",
                history=self.memory,
            )
            self.memory.append({"role": "assistant", "content": ai_response})

            if "修复代码建议" in ai_response:
                if self.human_in_the_loop_guard(file_name, ai_response):
                    print(f"已应用 {file_name} 的修复")

            self.files_processed_since_last_compaction += 1
            if self.files_processed_since_last_compaction >= 3:
                self.compact_memory()

if __name__ == "__main__":
    harness = SecurityHarness()
    harness.run("./src")

5.9 落地 Harness 的 4 条实战经验

  1. 别舍不得丢记忆。长任务失败 90% 是因为上下文太长导致模型“变傻”。Harness 的核心就是无情地阶段性清空历史，只留结构化摘要
  2. 对一切副作用上锁。任何涉及写数据库、发请求、删文件的工具，Harness 层面必须设人工白名单（Human-in-the-loop）——千万别盲目信任模型
  3. 提供结构化回退机制。Harness 发现死循环（连续 3 次报同一个错）就自动切断、记录快照、报错退出，不要一直重试烧钱
  4. 复杂任务拆成 Sub-Agents。不要用一个 Agent 从头走到尾——让 Harness 去调度“调研 Agent”、“写代码 Agent”、“测试 Agent”，它们交接时传递精简 JSON

5.10 从“调教 AI”到“工程化管理 AI”

掌握了 Harness，你会发现自己完成了一次范式升级：

  1. AI 的不可控性被消解——你不再担心 AI 把核心代码改坏，因为人工拦截像铁锁一样守住危险操作
  2. 上下文上限被终结——理论上只要有记忆压缩，这个 Harness 可以扫一万个文件不爆栈
  3. 能力高度模块化——把 security-check.md 换成 ui-design-guideline.md，同一个 Harness 立刻变成 UI 走查工具
核心范式：用代码控制边界（Harness），用大模型提供智力（Model），用文档规范标准（Skill）。
