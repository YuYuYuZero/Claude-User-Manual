第三章 · Claude Code 基础使用

3.1 Claude Code 是什么（先建立心智模型）

Claude Code 是一款命令行 AI 编程助手。它直接在终端运行，能读写文件、执行命令、理解整个代码仓库。

它不是编辑器插件——你不是在 VS Code 里用它，而是打开项目目录、启动它、用自然语言描述需求，它返回一个像同事一样会改代码、跑测试、写提交信息的 AI。

一句话区分：
  • 代码补全类工具：逐行补全你正在写的代码
  • Claude Code：给它任务，它自己决定读哪些文件、改哪些代码、跑什么命令——然后把结果报告给你

3.2 安装（macOS）

前置条件：macOS 10.15+、Node.js ≥ 18、可访问 AI 服务或合规 API 代理的网络。

3.2.1 装 Node.js（推荐用 nvm，避免 sudo 权限问题）

brew install nvm
mkdir -p ~/.nvm

# 写入 zsh 配置
echo 'export NVM_DIR="$HOME/.nvm"' >> ~/.zshrc
echo '[ -s "$(brew --prefix nvm)/nvm.sh" ] && . "$(brew --prefix nvm)/nvm.sh"' >> ~/.zshrc
source ~/.zshrc

# 安装 LTS 版本的 Node
nvm install --lts
node --version && npm --version

完成标志：终端输出类似 v22.x.x 的 Node 版本号 + 10.x.x 的 npm 版本号。

3.2.2 装 Claude Code CLI

npm install -g @anthropic-ai/claude-code
claude --version

完成标志：输出 @anthropic-ai/claude-code/x.y.z 格式的版本号。

⚠️ 如果你已经用 brew 装过 node，又遇到 EACCES 权限错：不要用 sudo 硬装——那样会埋下未来全局包冲突的雷。正确做法是先卸载 brew 版 node，再装 nvm: 
brew uninstall node
# 然后按 3.2.1 装 nvm 并重新装 LTS 版本

3.2.3 装 VS Code 扩展（可选但推荐）

在 VS Code 扩展市场搜索 Claude Code for VS Code（发布者：Anthropic），点击安装。装完重启 VS Code。

3.3 配置 API 代理

如果你用的是第三方 API 代理，需要修改 ~/.claude/settings.json。如果文件不存在：

mkdir -p ~/.claude && touch ~/.claude/settings.json

写入以下内容（把 你的 API Key 替换成真实的）:

{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "你的 API Key",
    "ANTHROPIC_BASE_URL": "https://api.你的代理.com",
    "ANTHROPIC_MODEL": "claude-opus-4-6",
    "ANTHROPIC_DEFAULT_OPUS_MODEL": "claude-opus-4-6",
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "claude-sonnet-4-6",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "claude-haiku-4-5-20251001",
    "API_TIMEOUT_MS": "3000000",
    "CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC": "1"
  },
  "autoUpdatesChannel": "latest"
}
关键易错点（务必看）:
  • ANTHROPIC_BASE_URL 在 Claude Code 里 不带 /v1——与直接调 REST API 不同
  • API_TIMEOUT_MS: 3000000 = 50 分钟，适合长任务；太短会在复杂任务中途断开
  • 配置完后必须重启终端 / VS Code，否则不生效

3.4 首次启动

进入你的项目目录
cd /path/to/your/project
启动 Claude Code
claude

首次启动会让你登录账号或配置 API 密钥，按提示走即可。

发送第一条测试指令

看到提示符 > 后，直接输入：

> 这个项目是做什么的?

完成标志：Claude 会读取项目里的 README 和主要文件，给你一段项目概述。

3.5 三个真实场景：它能做什么

讲再多概念不如看真实对话。下面是三种最高频的开发场景，都是简化后的真实交互流。

场景 A：修一个 bug

你:这个登录接口返回 500,帮我看看
Claude:[读代码][跑测试][定位到 auth.ts 第 47 行][修改][重新跑测试]
       已修复,原因是 JWT 过期时间判断反了,已加测试用例防止回归

场景 B：加新功能

你:给用户表加个软删除字段,别忘了更新所有查询
Claude:[读 schema][写迁移文件][grep 所有查询][逐个更新][跑测试]
       迁移文件已创建,12 处查询已更新,所有测试通过

场景 C：重构

你:这个 3000 行的文件太乱了,按职责拆成多个文件
Claude:[分析依赖][规划拆分][创建新文件][移动代码][更新 import][跑测试]
       已拆成 6 个文件,测试全绿
共同模式：你描述目标 → Claude 自己规划 + 自己动手 + 自己验证 → 给你结果。这是它和“代码补全”工具最根本的区别。

3.6 基本交互

启动后你会进入一个对话界面。直接用中文或英文描述需求：

> 这个项目是做什么的?
> 帮我把 login 接口的参数校验补上
> 运行一下单元测试
> 这段代码有什么问题?

Claude Code 会自己决定：读哪些文件、改哪些代码、执行什么命令。涉及文件修改或命令执行时，它会先展示“打算做什么”，等你确认再执行。

3.6.1 高频快捷键

按键作用
Enter发送
Shift + Enter换行（多行输入）
Ctrl + C中断当前操作（连按两次退出）
Ctrl + D退出程序
Esc Esc回滚到上一步（连按两次）
Tab显示 / 隐藏 Claude 的思考过程
Ctrl + R搜索历史命令
Shift + Tab循环切换“权限模式”（关键，见 §3.7）

3.6.2 三个魔法字符：@ # !

这三个字符会让 Claude Code 立刻进入特殊模式，效率提升明显：

字符用法例子
@<路径>文件路径自动补全——直接告诉 Claude 看哪个文件，比让它 grep 找省 token看看 @src/auth/login.ts 怎么实现的
!<命令>直接执行 bash，Claude 不介入。结果会自动进入对话上下文! cat error.log 然后 分析这个报错
#<文字>快速写入 memory，把临时结论沉淀到 CLAUDE.md# 项目用 pnpm 不是 npm
实用经验：养成 @文件路径 的习惯。让 Claude 自己 grep 找文件，它会读 5–8 个无关文件才定位，token 哗哗地烧；直接 @ 指路，一步到位。

3.6.3 读懂 Claude 的“工作日志”

Claude 干活时，屏幕上会输出一串“工具调用日志”。学会读它，你就能实时知道它在干什么：

* Read(package.json)
  └ Read 42 lines

* Bash(npm test)
  └ Running tests...
  └ All tests passed

* Edit(src/auth.ts)
  └ Updated 3 lines

关键符号：

  • * 表示 Claude调用了一个工具
  • └ 表示这个工具的执行结果
  • 工具名（Read / Edit / Bash / Glob / Grep 等）告诉你它在做什么
看到 Claude 在反复 Read 同一个文件，或在大量 Glob / Grep——它在“找东西”找不到。立刻 Ctrl+C 打断，直接用 @文件路径 告诉它去哪儿看。

3.6.4 日常必用的 10 个斜杠命令

命令作用什么时候用
/help列出所有命令忘记命令时的兜底
/clear清空当前对话切换到不相关的任务时（改完登录功能要改支付，先 /clear）
/compact [focus]压缩上下文，带 focus 参数告诉它保留什么上下文 > 80%，但还想继续当前任务
/context看上下文占用比例感觉 Claude 开始“遗忘”早期信息时
/cost查看本次会话花了多少钱避免一不小心烧了几十刀
/model切换模型（opus / sonnet / haiku）难题临时升 opus，完事切回 sonnet 省钱
/init为当前项目生成 CLAUDE.md进入任何新项目第一件事
/review对当前分支做代码审查Claude 改完一个东西，你眼花了让它自审
/permissions管理工具白名单把 npm test、git status 加白名单避免反复确认
/quit退出—

记忆技巧：不需要背，输入 / 就会有自动补全提示。

3.7 权限模式：Shift+Tab 循环切换（核心概念）

这是 Claude Code 最重要的概念。理解了权限模式，你就知道怎么在“安全”和“别打断我”之间平衡。

3.7.1 为什么要有权限模式

Claude Code 默认每个操作都问你：写文件问一次、跑 bash 命令问一次、访问网络问一次。

  • ✅ 好处：安全，不会乱动东西
  • ❌ 坏处：长任务会被打断几十次，烦到崩溃

所以官方设计了 5 种权限模式，你按需切换。

3.7.2 5 种权限模式

模式特点什么时候用
default每一步都问你不熟悉的代码库、关键生产代码、重要重构
acceptEdits自动批准：工作目录内的文件创建/编辑/删除、基础文件命令（mkdir/touch/rm/mv/cp/sed）。仍会问：npm test、git push、curl 等其他 bash 命令明确范围的改动，事后用 git diff 统一审
plan只读模式。可以读文件、提建议、出方案，但不改任何源码大重构之前，先让它出方案
auto分类器审查每一步：安全操作自动放行，危险操作自动拦截。需 Max+ 账号且 v2.1.83+长时间跑、不想被打扰
bypassPermissionsYOLO 模式：完全放行，不问不审，所有操作直接执行只在隔离环境用：Docker 容器、VM、沙盒
关于 acceptEdits 的常见误解：很多人以为它是“放行一切”，其实不是。它只放行文件编辑和基础文件操作，跑测试、推代码、调网络等仍会问你。
受保护路径永远会问：无论什么模式，对 .git/、.claude/、.claude/skills/ 的写入永远需要确认。这是 Claude Code 自己的“安全网”。

3.7.3 Shift+Tab 循环

按 Shift+Tab 在模式间切换，看屏幕底部状态栏会变：

default → acceptEdits → plan → [auto] → [bypassPermissions] → default
                                  ↑              ↑
                              账号资格够    启动时带了特定参数

默认只循环前 3 个。后 2 个要启用：

# 启用 auto mode(需 Max+ 账号)
claude --enable-auto-mode

# 让 bypassPermissions 进入循环(但不立即激活)
claude --allow-dangerously-skip-permissions

# 直接以 YOLO 模式启动(谨慎!)
claude --dangerously-skip-permissions

3.7.4 设置默认模式

不想每次手动切，在 ~/.claude/settings.json 里永久设置：

{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}

有效值：default、acceptEdits、plan、dontAsk、bypassPermissions。

3.7.5 给常用命令开白名单

想保持 default 模式的安全性，又不想每次 npm test 都问？用 /permissions 加白名单：

/permissions

# 添加 allow 规则
Bash(npm test)
Bash(git add *)
Bash(git commit *)
Bash(git status)

这样既保持默认模式的警觉，又不会被高频命令打扰。

3.7.6 推荐的模式选择

场景推荐模式
新项目第一次用、不了解代码库default
明确范围的改动，想事后用 git diff 统一审acceptEdits
大重构之前，先让它出方案plan
长时间跑、不想被打扰、有 Max 账号auto
CI/CD pipeline、Docker 容器内bypassPermissions

3.8 它走偏了怎么办：打断与改错恢复

3.8.1 症状：Claude 在往错的方向跑

解决：立刻按 Ctrl + C 打断它。然后重新描述需求：

> 等等,我不是要你重写整个文件,只要改那个 bug
不要等它继续——AI 一旦走偏，往往会越走越偏。早停早省钱。

3.8.2 症状：它改错了、改坏了

不要慌。按下面顺序处理：

  1. 按 Ctrl + C 打断当前操作
  2. 按两下 Esc——这是回滚快捷键，可以撤销 Claude 最近的改动
  3. 如果改动已经太多，在另一个终端窗口用 git 救：
git status              # 看被改了什么
git diff                # 看具体改了啥
git checkout <file>     # 单个文件撤销
git reset --hard        # 全部撤销,回到上次提交(慎用)

3.8.3 预防措施：养成“先 commit 再让它动手”的习惯

每次让 Claude 开始大改之前，先做一次 work-in-progress 提交（不用 push）:

git add .
git commit -m "wip: before claude refactor"
这个习惯能救命：无论 Claude 改坏了什么，一句 git reset --hard 就能回到上次提交的状态。没有 git，就没有后悔药——所以一开始就在有 git 的目录里用 Claude Code。

3.8.4 一个完整的“安全开始”流程

# 在一个还没初始化 git 的目录里,先初始化
git init
git add .
git commit -m "initial commit before claude code"

# 然后再启动 claude
claude

3.9 项目专属配置：CLAUDE.md

在项目根目录放一个 CLAUDE.md 文件，Claude Code 每次启动会自动读取。用来告诉它这个项目的专属规则：

# 项目规则

- 包管理工具用 pnpm,不要用 npm
- 提交信息写中文
- 测试框架是 vitest
- API 路由文件在 src/routes/ 目录下
- 代码规范遵循团队 eslint 配置

优先级：项目级 CLAUDE.md > 用户全局 ~/.claude/CLAUDE.md。

该文件建议提交到 Git，让团队成员共享同一套规则。新同事 clone 代码 + 装好 Claude Code，就自动拥有了团队的编码习惯。

3.9.1 一个完整的 CLAUDE.md 实例

下面这份 CLAUDE.md 来自一个真实的 FastAPI 项目，展示了一个“高质量”CLAUDE.md 应有的结构：

# 项目名称
这是一个基于 FastAPI 的订单管理服务。

## 技术栈
- Python 3.11 + FastAPI
- PostgreSQL 15 + SQLAlchemy 2.0 (async)
- Redis 用于缓存和消息队列
- pytest + pytest-asyncio 用于测试

## 目录结构
- `app/api/`      - HTTP 接口层
- `app/services/` - 业务逻辑层
- `app/models/`   - 数据库模型
- `app/utils/`    - 工具函数
- `tests/`        - 测试代码

## 开发规矩
- 所有数据库操作必须用 async/await
- 新增接口必须写集成测试,放在 tests/integration/
- 用 Pydantic v2 做数据校验,不要用 dataclasses
- 不要直接 print,用 logger (from app.utils.logging import logger)
- 所有金额字段用 Decimal,不要用 float

## 常用命令
- 跑所有测试:`pytest`
- 跑某模块测试:`pytest tests/unit/test_orders.py`
- 启动开发服务器:`uvicorn app.main:app --reload`
- 生成数据库迁移:`alembic revision --autogenerate -m "description"`

## 常见坑
- 本地跑测试前记得先启动 postgres 和 redis:`docker compose up -d`
- 不要直接修改 `alembic/versions/` 里的旧迁移,要改就加新迁移

## API 规范
详见 @docs/api-spec.md

3.9.2 CLAUDE.md 的 4 条核心原则

原则说明反例
保持在 200 行以内每次会话都加载它——越长每条消息的 token 成本就越高把整本 API 文档塞进 CLAUDE.md
写 Claude 容易犯的错例：“所有金额字段用 Decimal，不要用 float”——这样它就不会写出有精度问题的代码写“代码要规范”这种空话
写工具路径和命令例：“跑测试 pytest tests/”——它就不用自己摸索让 Claude 自己 grep 找 package.json
写目录约定例：“数据库模型放 app/models/”——新文件会放对位置不写，任由 Claude 自己决定文件放哪

3.9.3 多层 CLAUDE.md（个人偏好 + 项目规则）

CLAUDE.md 可以分层加载，优先级从下到上：

位置用途典型内容
~/.claude/CLAUDE.md个人全局——所有项目都加载语言偏好、注释风格、个人禁忌
<项目根>/CLAUDE.md项目级技术栈、目录约定、命令、坑
<子目录>/CLAUDE.md子目录级——只在这个子目录工作时加载子模块的特殊规则（如 frontend/ vs backend/）

个人全局的可以这样写：

# 个人偏好
- 代码注释用中文
- commit message 用中文,格式:<类型>(<范围>): <描述>
- 不要用 emoji
- 不要在回复里总结我让你做了什么,直接告诉我结果就行

3.9.4 用 @file 引用大文件

CLAUDE.md 里不要塞整个 API 文档，用引用就行：

## API 规范
详见 @docs/api-spec.md

Claude 需要的时候才会读 docs/api-spec.md，平时不占 token——这就是渐进式披露在 CLAUDE.md 上的应用。

💡 CLAUDE.md 何时生效：修改 CLAUDE.md 后，下次 claude 启动时才会重新加载；当前会话不受影响。如果你改完之后想立刻让新规则生效，需要 /quit 退出后重新 claude 启动（或者用 claude --resume 在新会话里接续原任务）。

3.10 终端命令透传（和 shell 配合）

对话里用 ! 前缀可以直接执行 shell 命令，结果自动进入对话上下文：

> ! cat error.log
> 帮我分析这个报错的原因并给出修复方案

也可以用管道做非交互模式（适合写进脚本）:

git diff | claude -p "帮我 review 这些改动"
cat server.log | claude -p "分析最近的服务异常"

其中 -p 表示 “prompt 模式”：输入一个问题、拿到回答后自动退出，不进入对话。

3.11 图片理解

直接把截图路径贴进对话：

> 看下这个页面截图有什么问题 /tmp/screenshot.png
> 根据这个设计稿生成前端代码 /Users/xxx/design.png

适合用来排查 UI 问题或让 AI 看设计稿生成代码。

3.12 典型工作流：排查并修复一个 Bug

> 运行 npm test 看看现在什么情况
(Claude 执行测试,发现 3 个用例失败)

> 分析一下失败原因,修一下
(Claude 读相关代码,定位问题,提出修改方案,等你确认)

> 再跑一遍测试
(确认全部通过)

> 看看当前改了什么,帮我写个 commit message

3.13 会话管理（长任务必看）

Claude Code 的对话有上下文窗口限制。当对话变长：

  • 手动压缩：输入 /compact，Claude 会总结之前的对话并释放空间
  • 清空重来：输入 /clear——当前话题已结束、要开新话题时用
  • 恢复会话：关掉终端后，下次用 claude --resume 或简写 claude -c 接续

3.14 配置文件一览

文件位置用途
CLAUDE.md项目根目录项目级指令——告诉 Claude 这个项目的规则
.claude/settings.json项目根目录项目级配置——权限白名单、MCP 服务器
.claude/commands/项目根目录项目级自定义 / 命令（团队共享，见第四章）
~/.claude/CLAUDE.md用户主目录全局指令——所有项目生效
~/.claude/settings.json用户主目录全局配置——API Key、默认模型、超时
~/.claude/commands/用户主目录个人级自定义 / 命令

项目级配置优先级 > 全局配置。
